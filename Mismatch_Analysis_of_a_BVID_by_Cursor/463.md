## 1. Executive Summary

This document presents a root-cause analysis of shadow-read mismatches observed during the migration of BVID 463's delivery-option generation from the legacy Order Service (OS) to the Retail Fulfillment Service (RFIS). 70 cleaned mismatch log files spanning March 3‚Äď12, 2026 were analyzed across 14 stores in multiple timezones.

The analysis surfaces **three primary root causes** and **two secondary patterns**:

| ID | Root Cause | Files Affected | Severity |
|----|-----------|---------------|----------|
| RC-1 | Legacy OS missing N days of schedule windows (calendar-day shift) | ~57 | **Critical** |
| RC-2 | RFIS pads future-day first window by `expressTimeMaxRange`; legacy uses store-open time directly | ~57 (embedded in RC-1 files) | **High** |
| RC-3 | `shouldInitiateFromNextHour` + `standardWindowUpperBound` causes today's scheduled windows to start dangerously late in legacy | ~10 | **High** |
| RC-4 | 20-min first-window offset on future days (RFIS 5-min schedule padding vs legacy raw store-open) | ~8 | Medium |
| RC-5 | Legacy midpoints retain sub-second precision; RFIS always truncates to the minute | ~20 | Low |

Additionally, **21 `no_count_mismatch` files** carry mismatches in standard-option fields (ETA range, eligibility, display strings) that are separate from window-count logic and require targeted follow-up.

> **Key finding**: The largest visible mismatch ‚ÄĒ legacy appearing to start scheduled windows 1‚Äď2 calendar days later than RFIS ‚ÄĒ is primarily caused by the legacy OS's MDS schedule-hour fetch omitting today (and sometimes tomorrow) from its `scheduleAvailableHours`, while RFIS includes them. Because `compareDeliveryAvailabilityResults` pairs days by **index** (not by date string), the comparison unknowingly matches RFIS day 0 (today) against legacy day 0 (day+2), producing artificial offsets of up to 2,890 minutes.

## 2. Detailed Mismatch Breakdown

### RC-1 ‚ÄĒ Calendar Day Shift: Legacy Missing N Days of Schedule Windows

#### 2.1.1 Observed Evidence

The most numerically dominant mismatch across the 70 files. Two sub-variants exist:

**RC-1A ‚ÄĒ 2-Day Shift** (stores `23323551`, `24323603`, `24323621`, `27482549`)

Request date is day D. RFIS generates schedule windows starting from day D. Legacy generates schedule windows starting from day D+2. Example from trace `13475821` (store `23323551`, request `2026-03-07T15:05:07Z`, timezone CST/CDT):

```
Day keyed as "2026-03-07":
  RFIS  first_midpoint : 2026-03-07T17:10:00Z     (11:10 AM CST ‚ÄĒ "today")
  Legacy first_midpoint: 2026-03-09T15:20:07.688714Z  (10:20 AM CDT ‚ÄĒ 2 days later!)
  start_offset_minutes : -2770.1  (‚Čą 46.2 hours)
  count_diff           : -2  (RFIS 23 windows vs Legacy 25)

Day keyed as "2026-03-08":
  RFIS  first_midpoint : 2026-03-08T16:00:00Z     (11:00 AM CDT)
  Legacy first_midpoint: 2026-03-10T14:20:00Z     (9:20 AM CDT)
  start_offset_minutes : -2780.0  (‚Čą 46.3 hours)
  count_diff           : -5  (RFIS 24 vs Legacy 29)

Days "2026-03-10", "2026-03-11":
  shadow only ‚ÄĒ no "original" entry (legacy ran out of days)
```

RFIS has 5 days (D through D+4); legacy has only 3 days (D+2 through D+4). The comparison pairs them positionally: RFIS[0]‚ÜĒLegacy[0], RFIS[1]‚ÜĒLegacy[1], RFIS[2]‚ÜĒLegacy[2]. The last 2 RFIS days have no legacy counterpart.

ETA for these stores: **110‚Äď120 min**. Sub-second precision observed in legacy midpoints (e.g., `15:20:07.688714Z`), confirming the legacy uses `currentTime` in its computation.

**RC-1B ‚ÄĒ 1-Day Shift** (stores `24388327`, `25019658`, `25019653`, `23350351`, `25157878`)

Same structure, but the gap is 1 day. Example from trace `3989a48b` (store `24388327`, request `2026-03-04T00:23:21Z`, timezone MST):

```
Day keyed as "2026-03-04":
  RFIS  first_midpoint : 2026-03-04T18:00:00Z   (11:00 AM MST ‚ÄĒ today)
  Legacy first_midpoint: 2026-03-05T16:20:00Z   (9:20 AM MST ‚ÄĒ tomorrow)
  start_offset_minutes : -1340.0  (= 22h20m)
  count_diff           : -5  (RFIS 21 vs Legacy 26)

Day "2026-03-07":
  shadow only ‚ÄĒ no "original" entry
```

#### 2.1.2 Root Cause Explanation

**The legacy `scheduleAvailableHours` received from MDS does not include today (and sometimes tomorrow) for these stores.** RFIS's `getScheduleStoreAvailabilityHours(sav)` parsing of the same SAV response returns more days, including today.

This means that when `compareDeliveryAvailabilityResults` calls `originalResult.availableDaysList.filterIndexed { index, availableDay ‚Üí availableDay.timeWindowsCount != shadowResult.availableDaysList[index].timeWindowsCount }` (index-based pairing), it compares:

- RFIS day[0] = today's windows ‚Üí paired against legacy day[0] = day+N's windows
- The resulting "mismatch" is a false comparison of two different calendar days

**Possible underlying causes** (requires runtime investigation to confirm one):
1. These stores have a minimum advance-booking requirement configured in MDS. MDS returns schedule availability only from day+N onward. The legacy consumes this directly; RFIS either parses it differently or uses ASAP hours to fill in the gap.
2. The legacy OS calls MDS with a different `requestedDays` base date (e.g., starting from tomorrow), while RFIS uses today as day 0.
3. RFIS's `getScheduleStoreAvailabilityHours` includes partial-day slots that the legacy's `scheduleAvailableHours` proto filtering excludes.

#### 2.1.3 Code Path

| System | Location | Relevant Line/Function |
|--------|----------|----------------------|
| Legacy OS | `DeliveryAvailabilityMapper.kt` | `mapScheduleAvailableDays` ‚ÄĒ iterates `deliveryAvailabilityResponseV2.scheduleAvailableHours` |
| RFIS | `GetBatchNewVerticalsDeliveryOptionDataHandler.kt` | `fetchStoreAvailability` ‚ÄĒ calls `merchantDataServiceStoreAvailabilityClient.batchGetConsumerStoreAvailability(storeIds, numDays, enableCache)` |
| RFIS | `ScheduledDeliveryOption.kt` | `buildDeliveryWindows` ‚ÄĒ iterates `scheduleStoreHours` from `deliveryOptionInfo.getStoreHourInstants(SCHEDULE)` |
| Comparison | `DeliveryOptionsUtil.kt` | Line 653: `originalResult.availableDaysList.filterIndexed { index, availableDay ‚Üí availableDay.timeWindowsCount != shadowResult.availableDaysList[index].timeWindowsCount }` ‚ÄĒ **index-based, not date-based** |

#### 2.1.4 Fix Required

1. **Investigate MDS discrepancy**: Add logging to compare `scheduleAvailableHours` count and start dates between the legacy response and what RFIS receives via `batchGetConsumerStoreAvailability` for the same store IDs. Confirm whether legacy and RFIS are getting different schedule day ranges.
2. **Fix RFIS or legacy to agree on day count**: If legacy is intentionally excluding today (e.g., no same-day scheduling for these stores), RFIS must be updated to do the same. If RFIS is correct, the legacy has a bug.
3. **Fix `compareDeliveryAvailabilityResults` to compare by date key, not by index**: The current `filterIndexed` logic produces false mismatches when the day count differs. It should pair days by their `availableDay.date` proto field, not by array position.

### RC-2 ‚ÄĒ ETA Padding on Future-Day First Windows: RFIS Starts Later, Fewer Windows Per Day

#### 2.2.1 Observed Evidence

On calendar days where BOTH systems generate windows (i.e., future days beyond the day-shift gap in RC-1), RFIS consistently starts the first window **~100 minutes later** than legacy, resulting in approximately 5 fewer windows per day. This is observed in the same stores as RC-1.

Example from trace `13475821` (store `23323551`, future day "2026-03-08", both on the same physical calendar day March 10):

```
RFIS  first window: 11:00 AM CDT = 16:00 UTC  ‚ÄĒ count: 24 windows
Legacy first window:  9:20 AM CDT = 14:20 UTC  ‚ÄĒ count: 29 windows
Offset: 16:00 - 14:20 = 100 minutes
Count diff: 24 - 29 = -5  (RFIS misses 5 windows at the beginning of the day)
```

This pattern holds consistently across all future days for high-ETA stores (ETA 110‚Äď120 min). The 100-minute difference ‚Čą `expressTimeMaxRange(120 min) - store_open_to_schedule_start_gap(20 min)`.

Conversely, for store `30869025` (ETA 32‚Äď47 min), future days show RFIS starting only 20 minutes **earlier** than legacy (count_diff = +1). This is explained below in RC-4.

#### 2.2.2 Root Cause Explanation

**RFIS pads the scheduled first-window start by the ASAP ETA on future days; legacy uses the raw store schedule open time.**

In RFIS `ScheduledDeliveryOption.buildDeliveryWindows` (approximately line 226‚Äď232):
```kotlin
val asapStartTimeWithPadding = deliveryOptionInfo.etaInfo.expressTimeMaxRange?.let {
    asapStartTime.plusMinutes(it.toLong())   // <‚ÄĒ adds full ETA to ASAP store open
} ?: asapStartTime

val scheduleStartTime = maxOf(zonedStartHour, deliveryOptionInfo.currentTime)
val scheduleStartTimeWithPadding = scheduleStartTime.plusMinutes(SCHEDULE_START_TIME_PADDING_MINUTES)  // +5 min

val combinedStartTime = maxOf(asapStartTimeWithPadding, scheduleStartTimeWithPadding)
```

When `asapStoreHours[index]` is available for the future day (i.e., `index < asapStoreHours.size`), `asapStartTime` = future day's ASAP store-open time. With ETA = 120 min, `asapStartTimeWithPadding` = store open + 120 min, dominating `combinedStartTime`. The first scheduled window is pushed ~100 min past store open.

In contrast, the legacy OS `mapScheduleAvailableDays` (line 3178 approx):
```kotlin
} else {
    storeDayOperationalHour.start   // <‚ÄĒ uses schedule open time directly, no ETA padding
}
```

The legacy uses `storeDayOperationalHour.start` as the first midpoint timestamp for non-MIC stores, with no ETA-based padding. This gives legacy an earlier start by ~100 min and ~5 extra windows per day.

#### 2.2.3 Code Path

| System | Location | Key Code |
|--------|----------|---------|
| RFIS | `ScheduledDeliveryOption.kt` ~line 226 | `asapStartTimeWithPadding = asapStartTime.plusMinutes(expressTimeMaxRange)` |
| RFIS | `ScheduledDeliveryOption.kt` ~line 228 | `combinedStartTime = maxOf(asapStartTimeWithPadding, scheduleStartTimeWithPadding)` |
| Legacy OS | `DeliveryAvailabilityMapper.kt` ~line 3178 | `midpointTimestamp = storeDayOperationalHour.start` (for non-MIC stores) |

#### 2.2.4 Fix Required

**RFIS should NOT apply ETA padding for future (non-today) scheduled days.** The ETA offset is meaningful on *today* to ensure the first scheduled window is after the current ASAP delivery. For future days, the first window should be at the store's schedule open time (matching legacy behavior):

```kotlin
// Proposed change in ScheduledDeliveryOption.buildDeliveryWindows:
val asapStartTimeWithPadding = if (isToday) {
    deliveryOptionInfo.etaInfo.expressTimeMaxRange?.let {
        asapStartTime.plusMinutes(it.toLong())
    } ?: asapStartTime
} else {
    asapStartTime  // no ETA padding for future days
}
```

This eliminates the ~100-min gap and ~5-window deficit on future days for high-ETA stores.

### RC-3 ‚ÄĒ `shouldInitiateFromNextHour` + `standardWindowUpperBound` Truncates Today's Windows

#### 2.3.1 Observed Evidence

For stores `30869025` and `24388225`, today's scheduled windows in legacy are severely truncated ‚ÄĒ showing only 3‚Äď5 windows starting at 6:00‚Äď6:20 PM local, while RFIS generates a full day (21‚Äď23 windows starting at 11:30 AM‚Äď1:00 PM local). The sub-second precision in legacy's today midpoints confirms the first window is derived from `currentTime`, not the store schedule open time.

**Store `30869025`** (8 files, request `2026-03-11T22:03:45.379Z`, timezone EDT UTC-4):

```
RFIS today ("2026-03-12"):
  first_midpoint : 2026-03-12T15:40:00Z    (11:30 AM EDT)
  window_count   : 23
  last_midpoint  : 2026-03-12T23:00:00Z    (7:00 PM EDT)

Legacy today ("2026-03-12"):
  first_midpoint : 2026-03-12T22:20:45.328336Z  (6:10 PM EDT)  ‚Üź sub-second!
  window_count   : 3
  last_midpoint  : 2026-03-12T23:00:45.328336Z  (7:00 PM EDT)

start_offset_minutes (today): -400.76  (‚Čą 6h41m ‚ÄĒ RFIS is 6h41m earlier)
count_diff (today)          : +20  (RFIS has 20 MORE windows today)
```

The sub-second part `0.328336` closely tracks the request's `0.379582` seconds, confirming the midpoint is `currentTime`-derived. Across all 8 files for this store, the sub-second part varies with each request but always produces only 3‚Äď5 windows late in the day. Future days consistently show a -20 min offset (discussed in RC-4).

**Store `24388225`** (trace `91792611`, request `2026-03-09T21:44:48.947Z`, timezone UTC-6):

```
RFIS today ("2026-03-10"):
  first_midpoint: 2026-03-10T17:00:00Z    (11:00 AM UTC-6)
  window_count  : 21

Legacy today ("2026-03-10"):
  first_midpoint: 2026-03-10T22:00:48.910379Z  (4:00 PM UTC-6)  ‚Üź sub-second!
  window_count  : 5
  count_diff    : +16 (RFIS has 16 more windows)

Future days ("2026-03-11" through "2026-03-13"):
  Legacy first_midpoint: 15:20 UTC (9:20 AM UTC-6)
  RFIS  first_midpoint: 17:00 UTC (11:00 AM UTC-6)
  start_offset_minutes: +100 (legacy 100 min EARLIER ‚ÄĒ same RC-2 pattern)
  count_diff: -5 (legacy has 5 more windows per future day)
```

#### 2.3.2 Root Cause Explanation

**The legacy OS's `shouldInitiateFromNextHour` experiment, when active, uses `standardWindowUpperBound` as the first scheduled-window starting point.** For stores 30869025 and 24388225, the `standardWindowUpperBound` (derived from `currentTime + ETA + standard_window_duration`) lands very late in the day ‚ÄĒ around 6:00‚Äď6:20 PM local ‚ÄĒ leaving only 3‚Äď5 remaining schedule windows before store close.

In legacy `DeliveryAvailabilityMapper.mapScheduleAvailableDays` (approximately lines 3162‚Äď3209):

```kotlin
val midpointTimestamp = if (isProjectMic) {
    if (shouldInitiateFromNextHour(eligibleForScheduleWindowFromNextHour,
                                   standardWindowUpperBound,
                                   storeDayOperationalHour)) {
        standardWindowUpperBound!!          // <‚ÄĒ takes the standard window upper bound
    } else if (currentTime.plus(ASAP_TIME_DURATION_HOURS, ChronoUnit.HOURS)
               > storeDayOperationalHour.start) {
        currentTime.plus(ASAP_TIME_DURATION_HOURS, ChronoUnit.HOURS)
    } else {
        storeDayOperationalHour.start
    }
} else { ... }
```

`shouldInitiateFromNextHour` returns `true` when:
- `eligibleForScheduleWindowFromNextHour` DV is **on**
- `standardWindowUpperBound != null`
- `standardWindowUpperBound > storeDayOperationalHour.start`

For these stores, all three conditions are met. The `standardWindowUpperBound` is computed as `currentTime + ETA + standard_window_size` ‚Čą late afternoon/early evening. This becomes the first scheduled midpoint for today, skipping the entire morning and afternoon.

RFIS has **no equivalent behavior**. Its `getRoundedUpMidpointTimestamp` does not consult `standardWindowUpperBound`. For today, RFIS computes `combinedStartTime = max(asapStartTimeWithPadding, scheduleStartTimeWithPadding)` which, when `scheduleStoreHours[0]` covers a future day (tomorrow), uses `scheduleStartTimeWithPadding = max(scheduleOpen, currentTime) + 5 min` ‚Üí a much earlier start.

#### 2.3.3 Code Path

| System | Location | Key Code |
|--------|----------|---------|
| Legacy OS | `DeliveryAvailabilityMapper.kt` ~lines 3162‚Äď3175 | `midpointTimestamp = standardWindowUpperBound!!` when `shouldInitiateFromNextHour = true` |
| Legacy OS | `DeliveryAvailabilityMapper.kt` ~line 3192 | `shouldInitiateFromNextHour(eligibleForScheduleWindowFromNextHour, standardWindowUpperBound, storeDayOperationalHour)` |
| RFIS | `ScheduledDeliveryOption.kt` ~line 310 | `getRoundedUpMidpointTimestamp` ‚ÄĒ uses `combinedStartTime`, no `standardWindowUpperBound` path |

**Dynamic Value involved**: `eligibleForScheduleWindowFromNextHour` ‚ÄĒ active for stores 30869025 and 24388225. If this DV is turned off, legacy would fall through to `currentTime + 2h` or `storeDayOperationalHour.start`, producing different (but still potentially mismatching) results.

#### 2.3.4 Fix Required

RFIS needs to replicate the `shouldInitiateFromNextHour` behavior OR the legacy DV needs to be disabled for BVID 463 stores before rollout.

**Option A (Recommended ‚ÄĒ implement in RFIS)**: Add a `combinedStartTime` floor at `max(combinedStartTime, standardWindowUpperBound)` when `eligibleForScheduleWindowFromNextHour` is active, so RFIS's today-first-window also starts no earlier than the standard window's upper bound.

**Option B**: Disable the `eligibleForScheduleWindowFromNextHour` DV for BVID 463 stores in the legacy, confirm parity with RFIS, then re-enable after RFIS implements Option A.

### RC-4 ‚ÄĒ 20-Minute First-Window Offset on Future Days (RFIS Earlier)

#### 2.4.1 Observed Evidence

For store `30869025` (ETA 32‚Äď47 min), all 4 future days consistently show RFIS starting **20 minutes earlier** than legacy:

```
Future day "2026-03-13":
  RFIS  first_midpoint: 2026-03-13T15:40:00Z  (11:30 AM EDT)  ‚Üí midpoint 11:40 AM
  Legacy first_midpoint: 2026-03-13T16:00:00Z  (12:00 PM EDT)
  start_offset_minutes : -20.0  (consistent across all 4 future days)
  count_diff           : +1  (RFIS has 1 more window per future day)
```

One request has -30 min on future days (trace `59409e7f`), suggesting a 10-min rounding boundary effect.

#### 2.4.2 Root Cause Explanation

For future days where `index >= asapStoreHours.size` (i.e., RFIS only has ASAP hours for today, not for all future days), RFIS falls back to `asapStartTime = currentTime` (past timestamp):

```kotlin
val asapStartTime = if (index < asapStoreHours.size) {
    maxOf(asapStoreHours[index].first.atZone(tz), currentTime)
} else {
    currentTime    // <‚ÄĒ falls back to current time for out-of-range indices
}
```

When `asapStartTime = currentTime` (which is in the past relative to the future day), `asapStartTimeWithPadding = currentTime + ETA` is also in the past. Thus `combinedStartTime = scheduleStartTimeWithPadding = storeOpen + 5 min`. After 10-min rounding: `storeOpen + 5 min ‚Üí rounds up to storeOpen + (10 - r) = storeOpen + N`, producing a first midpoint that is 5‚Äď15 min after store open.

The legacy, on the other hand, uses `storeDayOperationalHour.start` directly (e.g., 12:00 PM EDT = store open + some offset), ending up 20 min later.

The 20-min difference = legacy store schedule open (16:00 UTC = 12:00 PM EDT) vs RFIS schedule open + 5-min padding rounded to nearest 10 (15:30 + 5 = 15:35 ‚Üí rounds to 15:40 UTC = 11:40 AM EDT, start 11:30 AM).

#### 2.4.3 Code Path

| System | Location | Key Code |
|--------|----------|---------|
| RFIS | `ScheduledDeliveryOption.kt` ~line 219 | `asapStartTime = currentTime` when `index >= asapStoreHours.size` |
| RFIS | `ScheduledDeliveryOption.kt` ~line 229 | `scheduleStartTimeWithPadding = scheduleStartTime.plusMinutes(5L)` ‚Üí drives `combinedStartTime` |
| Legacy OS | `DeliveryAvailabilityMapper.kt` ~line 3178 | `storeDayOperationalHour.start` used directly |

#### 2.4.4 Fix Required

Ensure `asapStoreHours` covers the same number of days as `scheduleStoreHours`. If MDS returns ASAP availability for only 1 day but schedule availability for 4 days, RFIS should either:
- Request `numDays` worth of ASAP hours to cover all schedule days, OR
- Use the last available ASAP day's open time (rather than `currentTime`) for out-of-range indices

This would make RFIS future-day first windows consistent across all days regardless of how many ASAP-hour days MDS returns.

### RC-5 ‚ÄĒ Sub-Second Precision in Legacy Midpoints (No Truncation)

#### 2.5.1 Observed Evidence

Legacy midpoints carry microsecond precision inherited from `currentTime`, while RFIS always truncates to the minute:

```
Legacy: 2026-03-09T15:20:07.688714Z   ‚Üź microseconds from request time
RFIS:   2026-03-07T17:10:00Z           ‚Üź clean minute boundary
```

Sub-second legacy midpoints confirmed in 20+ log files across stores `23323551`, `24323603`, `24323621`, `23350351`, `24388225`, `30869025` (all RC-1A and RC-3 stores). Stores `24388327`, `25019658`, `25019653` show clean legacy midpoints (sub-second DV appears off for those stores).

#### 2.5.2 Root Cause Explanation

The legacy OS conditionally truncates midpoint timestamps based on the `shouldTruncateWindowHours` dynamic value. When the DV is **off**, legacy propagates the sub-second part of `currentTime` through its arithmetic without ever calling `.truncatedTo(ChronoUnit.MINUTES)`.

RFIS's `TimeHelper.getRoundedUpTruncatedTime` **always** truncates:

```kotlin
// TimeHelper.kt ~line 223
fun getRoundedUpTruncatedTime(time: ZonedDateTime, roundUpMinutes: Int): ZonedDateTime {
    return if (time.minute.rem(roundUpMinutes) != 0) {
        time.plus(roundUpMinutes - time.minute.rem(roundUpMinutes).toLong(), MINUTES)
            .truncatedTo(MINUTES)   // <‚ÄĒ always truncates
    } else {
        time.truncatedTo(MINUTES)   // <‚ÄĒ always truncates
    }
}
```

In contrast, the legacy at `DeliveryAvailabilityMapper.kt` ~line 3186‚Äď3188:
```kotlin
val midpointTimestampRoundedUp = if (shouldTruncateWindowHours) {
    midpointTimestampRoundedUpPreTruncate.truncatedTo(ChronoUnit.MINUTES)
} else {
    midpointTimestampRoundedUpPreTruncate    // <‚ÄĒ no truncation when DV is off
}
```

#### 2.5.3 Impact

The `compareDeliveryAvailabilityResults` comparison does **not** compare midpoint timestamps (`availableDays` check is count-only). Sub-second precision does not itself trigger a proto comparison mismatch, but it:
- Makes offset calculations in the Python analysis script appear non-integer
- Creates inconsistency if any downstream consumer compares timestamps for equality
- Means `shouldTruncateWindowHours` must be confirmed ON before RFIS go-live to avoid user-visible display jitter

#### 2.5.4 Fix Required

Enable the `shouldTruncateWindowHours` DV for all BVID 463 stores. This is a pre-flight checklist item, not a code change.

### RC-6 ‚ÄĒ `no_count_mismatch` Files: Standard Option / ETA Discrepancies

#### 2.6.1 Observed Evidence

21 files carry pattern `no_count_mismatch`. The Python comparison script found no window-count differences, but the OS shadow-read comparison did catch a mismatch (otherwise these would not be in the mismatch event stream). Key observations:

- Stores `25157878` (2 files): `standard_elig = null` in RFIS vs presumably `true` in legacy ‚Üí `asapAvailable` proto field mismatch
- Stores `23350351` (6 files): Window counts match (same count despite 1-day date shift), but `asapMinutesRange` likely differs ‚Üí ETA display mismatch
- Stores `23323551`, `24388327`, `27482549`, `24388225`, `25019653` also appear in no_count_mismatch ‚ÄĒ these are day-shift stores where window counts coincidentally match but content (times, display strings) differs
- Stores `24323597`, `24362200`, `28102267`: clean stores with no day-shift; mismatches are in standard-option fields only

These 21 files represent the "standard ETA / eligibility" mismatch class, which is independent of the scheduled window generation issues above. The `compareDeliveryAvailabilityResults` checks `asapMinutesRange` with ¬Ī1 min tolerance; any difference beyond ¬Ī1 triggers a mismatch.

#### 2.6.2 Likely Root Causes

1. **`asapAvailable` mismatch**: RFIS sets `eligibility = null` (maps to proto false) for some stores where legacy returns `asapAvailable = true`. This may be due to RFIS's standard-option eligibility criteria differing from the legacy check.
2. **`asapMinutesRange` boundary**: RFIS uses `expressTimeMaxRange + 1` in `StandardDeliveryOption.getStandardStartTime` (RFIS `StandardDeliveryOption.kt` ~line 308), adding an extra minute that legacy doesn't add. For stores near the ¬Ī1 min tolerance boundary this causes a mismatch.
3. **ETA display string format**: Some stores show `"109‚Äď119 min"` in RFIS vs `"110‚Äď120 min"` in legacy (trace `9d23f83c`, `b99dedbd`), suggesting a 1-minute difference in the computed range.

#### 2.6.3 Fix Required

Investigate each `no_count_mismatch` sub-cluster:
- For `standard_elig = null` stores: validate RFIS eligibility logic against the legacy `setStandardLegacyDeliveryOptionData` path.
- For ETA display mismatches: review the `+1 minute` in `StandardDeliveryOption.getStandardStartTime` and confirm intentionality vs parity gap.

## 5. What Needs To Be Done

### Fix 1 ‚ÄĒ Align Schedule Day Count Between RFIS and Legacy (RC-1)

**Priority: Critical (blocks migration)**

**Problem**: The legacy `scheduleAvailableHours` covers N fewer days than RFIS's `scheduleStoreHours`. The comparison's index-based pairing (`filterIndexed`) then matches the wrong calendar days.

**Action Items**:
1. Add debug logging to the shadow-read path to print `scheduleAvailableHours.size` (legacy) and `scheduleStoreHours.size` (RFIS) for the same request. Compare day-start timestamps.
2. If legacy is correctly excluding today due to a store-level minimum-advance-booking policy, update RFIS's `getScheduleStoreAvailabilityHours` to apply the same exclusion. Look for a `minimumDaysInAdvance` field in the SAV response proto.
3. If legacy is incorrectly excluding today, file a legacy bug and confirm RFIS behavior is correct.
4. Fix `compareDeliveryAvailabilityResults` in `DeliveryOptionsUtil.kt` to pair days by date key, not by array index:
   ```kotlin
   // Replace filterIndexed with a map-based approach:
   val originalDaysByDate = originalResult.availableDaysList.associateBy { it.date }
   val differences = shadowResult.availableDaysList.filter { shadowDay ->
       val originalDay = originalDaysByDate[shadowDay.date]
       originalDay != null && shadowDay.timeWindowsCount != originalDay.timeWindowsCount
   }
   ```

### Fix 2 ‚ÄĒ Remove ETA Padding on Future Schedule Days in RFIS (RC-2)

**Priority: High**

**Problem**: RFIS pushes future-day first windows by `expressTimeMaxRange` minutes, giving ~5 fewer windows per day vs legacy for high-ETA stores.

**Before** (`ScheduledDeliveryOption.kt` ~line 226):
```kotlin
val asapStartTimeWithPadding = deliveryOptionInfo.etaInfo.expressTimeMaxRange?.let {
    asapStartTime.plusMinutes(it.toLong())
} ?: asapStartTime
```

**After**:
```kotlin
val asapStartTimeWithPadding = if (isCurrentDayOrBeforeStoreOpen) {
    deliveryOptionInfo.etaInfo.expressTimeMaxRange?.let {
        asapStartTime.plusMinutes(it.toLong())
    } ?: asapStartTime
} else {
    asapStartTime  // future days: no ETA padding, start from store open time
}
```

where `isCurrentDayOrBeforeStoreOpen = currentTime.isBefore(zonedStartHour)` (request time is before today's store opens, or this is a future day).

### Fix 3 ‚ÄĒ Implement `shouldInitiateFromNextHour` Parity in RFIS (RC-3)

**Priority: High**

**Problem**: For stores with `eligibleForScheduleWindowFromNextHour` active, legacy uses `standardWindowUpperBound` as the today-first-window anchor, cutting off most of the day. RFIS always starts from `combinedStartTime`, which is much earlier.

**Before/After** (RFIS `ScheduledDeliveryOption.kt` ~line 310 in `getRoundedUpMidpointTimestamp`):

```kotlin
// Current RFIS behavior ‚ÄĒ no standardWindowUpperBound check
var firstMidpoint = combinedStartTime

// Proposed addition
val standardUpperBound = deliveryOptionInfo.standardWindowUpperBound
if (isToday
    && deliveryOptionInfo.isEligibleForScheduleWindowFromNextHour()
    && standardUpperBound != null
    && standardUpperBound.isAfter(firstMidpoint)) {
    firstMidpoint = standardUpperBound.atZone(deliveryOptionInfo.timezone)
}
```

**Note**: This requires `deliveryOptionInfo` to carry `standardWindowUpperBound` and the `isEligibleForScheduleWindowFromNextHour` flag. These must be propagated from the RFIS handler through the `DeliveryOptionInfo` model.

### Fix 4 ‚ÄĒ Use Last Available ASAP Day for Out-of-Range Future Indices (RC-4)

**Priority: Medium**

**Problem**: When `index >= asapStoreHours.size`, RFIS falls back to `currentTime` as `asapStartTime`, producing a future-day first window 20 min earlier than legacy.

**Before** (`ScheduledDeliveryOption.kt` ~line 219):
```kotlin
val asapStartTime = if (index < asapStoreHours.size) {
    maxOf(asapStoreHours[index].first.atZone(tz), currentTime)
} else {
    currentTime
}
```

**After**:
```kotlin
val asapStartTime = if (index < asapStoreHours.size) {
    maxOf(asapStoreHours[index].first.atZone(tz), currentTime)
} else if (asapStoreHours.isNotEmpty()) {
    // Use the last known ASAP open time (same store hours apply)
    asapStoreHours.last().first.atZone(tz)
} else {
    currentTime
}
```

This ensures future-day first windows start at the correct store-open time rather than creating an artificial offset from `currentTime`.

### Fix 5 ‚ÄĒ Enable `shouldTruncateWindowHours` DV (RC-5)

**Priority: Medium (pre-flight checklist)**

Enable `shouldTruncateWindowHours = true` for all BVID 463 stores in the legacy OS before go-live. This removes sub-second precision from legacy midpoints, ensuring clean minute-boundary timestamps that match RFIS output.

Verify with a targeted shadow-read after DV change before promoting the BVID.

### Fix 6 ‚ÄĒ Investigate Standard Option ETA / Eligibility Parity (RC-6)

**Priority: Medium**

For the 21 `no_count_mismatch` files:
1. Review `StandardDeliveryOption.getStandardStartTime` (`+1 min` addition for non-DmMic path at ~line 308) ‚ÄĒ confirm this is intentional or a parity gap.
2. For `standard_elig = null` stores (25157878): trace the RFIS standard-option eligibility path and compare against legacy `setStandardLegacyDeliveryOptionData`. Identify which condition causes RFIS to return `null` eligibility.
3. Add explicit ETA tolerance/normalization in `compareDeliveryAvailabilityResults` or align the computation to remove the 1-min boundary sensitivity.

## 6. Conclusion

BVID 463's shadow-read mismatches fall primarily into a **schedule window availability gap between legacy OS and RFIS**: legacy omits 1‚Äď2 days of schedule windows that RFIS includes (RC-1), RFIS pads future-day windows by the full ETA giving fewer windows per day (RC-2), and for a subset of stores a legacy DV (`eligibleForScheduleWindowFromNextHour`) severely truncates today's schedule options by anchoring to `standardWindowUpperBound` late in the day (RC-3).

The `compareDeliveryAvailabilityResults` function's **index-based day pairing** compounds RC-1 by producing artificial "1-2 day time offsets" that look like timestamp bugs but are really misaligned day arrays. Fixing the comparison to pair by date key would surface the true mismatch (RC-1: missing days) more accurately and reduce noise.

The three critical fixes ‚ÄĒ aligning schedule day range (Fix 1), removing ETA padding on future days (Fix 2), and implementing `shouldInitiateFromNextHour` parity (Fix 3) ‚ÄĒ are prerequisites for a clean migration of BVID 463. The remaining fixes (RC-4, RC-5, RC-6) are lower risk but should be resolved before graduated rollout to eliminate residual mismatch noise in monitoring dashboards.

Once these fixes are in place, re-running the shadow read for a representative set of the 14 stores should reduce the mismatch event count to near zero for scheduled windows, with only potential display-string differences remaining.
