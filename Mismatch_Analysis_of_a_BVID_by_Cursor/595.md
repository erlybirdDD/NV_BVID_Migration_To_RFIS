#### 3.2.2 Root Cause Explanation

**Two compounding factors produce the consistent 10-minute gap:**

**Factor A — RFIS schedule hours start earlier than legacy.**  
The RFIS `getScheduleStoreAvailabilityHours(sav)` returns a schedule start of approximately `22:15Z` (9:15 AM AEDT) for weekdays, while the legacy's `storeDayOperationalHour.start = 22:30Z` (9:30 AM AEDT) for the same days. This 15-minute difference in how the same MDS SAV response is parsed drives the discrepancy.

**Factor B — RFIS applies `SCHEDULE_START_TIME_PADDING_MINUTES = 5` then rounds up to nearest 10.**  
For future days where `index >= asapStoreHours.size` (RFIS only has ASAP hours for today, not all 5 future days), `asapStartTime = currentTime` (past timestamp), so `combinedStartTime` is dominated by `scheduleStartTimeWithPadding`:

```kotlin
// ScheduledDeliveryOption.kt ~line 219–230
val asapStartTime = if (index < asapStoreHours.size) {
    maxOf(asapStoreHours[index].first.atZone(tz), currentTime)
} else {
    currentTime  // <— falls back to past timestamp for future day indices
}
val asapStartTimeWithPadding = asapStartTime.plusMinutes(expressTimeMaxRange)  // still in the past
val scheduleStartTimeWithPadding = scheduleStartTime.plusMinutes(5L)  // = 22:15 + 5 = 22:20Z
val combinedStartTime = maxOf(asapStartTimeWithPadding, scheduleStartTimeWithPadding)
// = 22:20Z (schedule dominates since asap is in the past)
```

`getRoundedUpTruncatedTime(22:20, 10)`: `20 % 10 = 0` → `22:20Z`. **RFIS first midpoint = `22:20Z`** = 9:20 AM AEDT ✓

The legacy OS uses `storeDayOperationalHour.start` directly:
- `storeDayOperationalHour.start = 22:30Z` (9:30 AM AEDT)
- `22:30 % 10 = 0` → no rounding needed → `midpointTimestampRoundedUp = 22:30Z`
- `isNonGroceryNewVerticalsWindowSizeChangeEnabled = false`, `shouldInitiateFromNextHour = false` → **Legacy first midpoint = `22:30Z`** = 9:30 AM AEDT ✓

**Result: RFIS consistently 10 min earlier = +1 window on most days.**

The clean timestamps (no sub-second) on both sides confirm these are properly derived from store schedule hours, not from `currentTime`.

## 6. What Needs To Be Done

### Fix 1 — Investigate `ASAP_TIME_DURATION_HOURS` DV Alignment for BVID 595 (RC-1)

**Priority: Medium**

The legacy's `ASAP_TIME_DURATION_HOURS` (or a BVID/store-level DV override) appears to be **1 hour** for this store rather than the standard 2 hours. This causes today's first scheduled window to start at `currentTime + 1h` instead of `currentTime + 2h`.

Confirm the DV value for BVID 595. If it is intentionally 1h (e.g., express-delivery store type), RFIS must replicate this configuration. Currently RFIS always uses `expressTimeMaxRange` from the ETA response for its `asapStartTimeWithPadding`, which produces a different result. Add a BVID/store-level configuration to RFIS for the "minimum advance time for today's first scheduled window."

**Before RFIS go-live**: verify that the RFIS `combinedStartTime` for today produces the same first-window time as the legacy for this store under a range of request times.

### Fix 2 — Align Schedule Store Hours Parsing (RC-2)

**Priority: High**

RFIS's `getScheduleStoreAvailabilityHours(sav)` returns a schedule start ~15 min earlier (9:15 AM AEDT) than legacy's `scheduleAvailableHours` (9:30 AM AEDT) for the same store. This causes RFIS to generate 1 extra future-day window.

**Action**: Add debug logging to compare `scheduleAvailableHours[0].start` (legacy) vs `scheduleStoreHours[0].first` (RFIS) for store `2460373`. Identify whether the difference is in:
- The MDS SAV response proto itself
- How legacy vs RFIS parse the `storeAvailability.scheduleHours` field
- A clock-alignment artifact (e.g., legacy rounds to 10-min boundaries during parsing, RFIS doesn't)

If RFIS is fetching a different granularity of store hours, align the parsing to match legacy behavior.

### Fix 3 — Fix `asapStoreHours` Out-of-Range Fallback (RC-2)

**Priority: Medium**

When `index >= asapStoreHours.size`, RFIS uses `currentTime` as `asapStartTime`. For this store, `asapStoreHours.size = 1` (only covers today), so all future days fall back to `currentTime`. This is the same RC-4 identified in the BVID 463 analysis.

**Before** (`ScheduledDeliveryOption.kt` ~line 219):
```kotlin
val asapStartTime = if (index < asapStoreHours.size) {
    maxOf(asapStoreHours[index].first.atZone(tz), currentTime)
} else {
    currentTime  // <— produces past timestamp, combinedStartTime driven by schedule
}
```

**After**:
```kotlin
val asapStartTime = if (index < asapStoreHours.size) {
    maxOf(asapStoreHours[index].first.atZone(tz), currentTime)
} else if (asapStoreHours.isNotEmpty()) {
    asapStoreHours.last().first.atZone(tz)  // use last known ASAP open time
} else {
    currentTime
}
```

This ensures future days use the store's ASAP open time (not `currentTime`) as the ASAP anchor, producing a consistent first-window time aligned with the schedule opening.

### Fix 4 — Enable `shouldTruncateWindowHours` DV (RC-1b)

**Priority: Medium (pre-flight checklist)**

Enable `shouldTruncateWindowHours = true` for BVID 595 stores to eliminate sub-second precision in legacy midpoints. This removes the `"1:50 pm-2:10 pm"` vs `"2:00 pm-2:20 pm"` residual display mismatch caused by sub-second precision on today's first window and aligns timestamps to minute boundaries, matching RFIS output.

## 7. Conclusion

BVID 595's shadow-read mismatches are driven by two clean, well-scoped root causes affecting a single Australian store. Unlike BVID 463, there is **no calendar day shift** — both systems generate the same number of schedule days. The mismatches are purely in the timing of the first window.

**RC-1** (today: ~9 min gap) is a consequence of the legacy OS using `currentTime + ASAP_TIME_DURATION_HOURS` (configured to ~1h for this store) as today's scheduled window anchor, while RFIS applies a 60-minute round-up plus half-interval via the `isToday` code path. The result is RFIS's first today-window starting 9 minutes later than legacy with a different display string. Window counts today are identical.

**RC-2** (future days: consistent −10 min, +1 window) is caused by RFIS parsing schedule hours ~15 min earlier than legacy and applying a 5-minute padding before rounding, producing a first future-day window that is systematically 10 minutes earlier. This gives RFIS exactly 1 extra window per future day on most weekdays and some weekends.

Once Fixes 1–4 are applied and the shadow read re-run, all four files should reduce to `no_count_mismatch` with at most a sub-second residual on today's midpoint (which Fix 4 eliminates). Neither root cause is BVID-specific architectural complexity — both are deterministic, per-store configuration issues that can be resolved through DV and parsing alignment.
