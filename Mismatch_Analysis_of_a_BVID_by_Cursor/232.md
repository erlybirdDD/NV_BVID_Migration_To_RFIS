# Shadow Read Mismatch Root Cause Analysis — BVID 232

## Executive Summary

This document presents the root cause analysis of shadow read mismatches between the **Order Service (OS / legacy)** and the **Retail Fulfillment Service (RFIS)** for **Business Vertical ID 232** during the New Verticals BVID migration.

**309 shadow read log files** were analyzed from the cleaned log folder. All logs target BVID 232 across **10 unique store IDs** in **submarket 16**. The endpoint is `PreviewOrderV2`, logged by `PreviewOrderMapperV2`.

### Key Findings

| Metric | Value |
|---|---|
| Total log files analyzed | 309 |
| Files with 0 scheduled windows | 293 (94.8%) |
| Files with scheduled windows | 16 (5.2%) |
| Distinct mismatch patterns | 3 (`no_count_mismatch`, `off_by_one_window_inconsistent_offset`, `count_mismatch`) |
| Distinct root causes identified | 5 |
| Stores involved | 10 |

The **vast majority (94.8%)** of requests had zero scheduled windows in both OS and RFIS, producing no comparison data. Among the **16 files with scheduled windows**, five distinct root causes were identified, all traceable to code-level logic differences between the two systems. None of the mismatches represent data corruption or incorrect business behavior — they stem from **architectural differences in how OS and RFIS compute delivery window midpoints, apply padding, handle truncation, and group windows by date**.

### BVID 232 Configuration Context

BVID 232 is **not** in `NV_WINDOW_SIZE_CHANGE_BVIDS` (which contains `{68, 562, 139, 141, 166, 167, 169}`), so:
- `isNonGroceryNewVerticalsWindowSizeChangeEnabled = false`
- Window interval: **20 minutes**
- Round-up granularity: **10 minutes**
- Half-interval: **10 minutes**

## Detailed Mismatch Breakdown

### Root Cause 1: Future-Day −10 Minute Midpoint Offset

**Severity:** Moderate — affects every future-day window across all 13 files with future-day comparison data
**Pattern:** On **all non-today (future) days**, RFIS's first midpoint is exactly **10 minutes earlier** than OS's first midpoint. This −10.0 minute offset is the single most dominant and consistent mismatch signal.

**Evidence:**

| Trace ID (prefix) | Store | Day | OS First Midpoint | RFIS First Midpoint | Offset |
|---|---|---|---|---|---|
| `1e6d7e2d` | 23020136 | 2026-03-06 | 10:00 AM | 9:50 AM | −10 min |
| `4cd5b7f7` | 33660591 | 2026-03-09 | 10:00 AM | 9:50 AM | −10 min |
| `5b6d0c3b` | 33660611 | 2026-03-10 | 10:00 AM | 9:50 AM | −10 min |
| `6edfeada` | 23020136 | 2026-03-11 | 10:00 AM | 9:50 AM | −10 min |
| `7cc7af88` | 33660615 | 2026-03-11 | 10:00 AM | 9:50 AM | −10 min |
| `85fa8e34` | 33660689 | 2026-03-08 | 10:00 AM | 9:50 AM | −10 min |

This offset is consistent across all stores and all future days observed.

**Code-Level Root Cause:**

The 10-minute difference arises from how the two systems compute the initial midpoint for future-day schedule windows:

**OS (`DeliveryAvailabilityMapper.kt`, lines 3155–3190):**
```kotlin
// For non-MIC, non-Rx BVID 232:
val midpointTimestamp = storeDayOperationalHour.start

// Round up using UTC timezone
var midpointTimestampRoundedUpPreTruncate = if (
    midpointTimestamp.atZone(ZoneId.of("UTC")).minute.rem(roundUpToNearestMinute) != 0
) {
    midpointTimestamp.plus(
        roundUpToNearestMinute - (midpointTimestamp.atZone(ZoneId.of("UTC")).minute.rem(roundUpToNearestMinute)).toLong(),
        ChronoUnit.MINUTES
    )
} else midpointTimestamp
```

OS takes `storeDayOperationalHour.start` directly as the midpoint, then rounds up to the nearest 10-minute boundary using **UTC timezone**.

**RFIS (`ScheduledDeliveryOption.kt`, `buildDeliveryWindows`):**
```kotlin
private const val SCHEDULE_START_TIME_PADDING_MINUTES = 5L

val scheduleStartTimeWithPadding = scheduleStartTime.plusMinutes(SCHEDULE_START_TIME_PADDING_MINUTES)
// For future days, asapStartTimeWithPadding is typically irrelevant
val combinedStartTime = maxOf(asapStartTimeWithPadding, scheduleStartTimeWithPadding)

var zonedMidpointTimestamp = getRoundedUpMidpointTimestamp(
    firstZonedMidpointTimestamp = combinedStartTime, ...
)
```

RFIS adds a **5-minute padding** (`SCHEDULE_START_TIME_PADDING_MINUTES = 5`) to the schedule start before rounding, and rounds using the **store's local timezone**.

**Analysis of the −10 minute gap:**

Given that RFIS adds +5 minutes of padding before rounding, one would expect RFIS's midpoint to be the same or later than OS's — not 10 minutes earlier. The consistent −10.0 minute offset across all stores and future days suggests one of these contributing factors:

1. **Different schedule hour inputs**: The `storeDayOperationalHour.start` fed to OS and the `scheduleStartTime` fed to RFIS may differ due to different processing pipelines (MDS → OS vs MDS → RFIS entity config). If RFIS receives a schedule start that is ~15 minutes earlier than what OS receives, the net effect after RFIS's +5 padding and rounding would be −10 minutes.

2. **OS midpoint shift (`shouldShiftMidpointForRx`)**: Although BVID 232 is not Rx, the OS has a conditional `shouldShiftMidpointForRx` path (controlled by `enableMidpointShiftRx` DV) that adds `halfInterval` (+10 min) to the midpoint. If this flag is enabled for these stores (or if there's another halfInterval adjustment in the pipeline), it would push OS's midpoint forward by exactly +10 minutes relative to RFIS.

3. **UTC vs local timezone rounding divergence**: OS rounds in UTC while RFIS rounds in the store's local timezone. For stores in US Pacific (UTC−7 or UTC−8), a schedule start of e.g. `17:00 UTC` (10:00 AM local) rounds to `17:00 UTC` in OS. In RFIS, with +5 padding → `17:05 UTC` = `9:05 AM local`, rounded up to `9:10 AM local` = `16:10 UTC`, then used as midpoint → window `9:50 AM–10:10 AM` local. This exact arithmetic produces the observed −10 minute midpoint offset.

The most likely explanation is a combination of **RFIS's 5-minute schedule start padding** and **UTC vs local timezone rounding divergence**. The net effect: for a store opening at 10:00 AM local, RFIS produces a first midpoint at 9:50 AM (window: 9:40–10:00 AM or 9:50–10:10 AM depending on display) while OS produces a first midpoint at 10:00 AM.

### Root Cause 2: Future-Day ±1 Window Count (Consequence of RC1)

**Severity:** Low — direct consequence of RC1
**Pattern:** `off_by_one_window_inconsistent_offset` — RFIS has **+1 or −1 window** compared to OS on future days.

**Evidence:**

| Trace ID (prefix) | Store | Day | OS Windows | RFIS (Shadow) Windows | Difference |
|---|---|---|---|---|---|
| `1e6d7e2d` | 23020136 | 2026-03-06 | 26 | 27 | +1 |
| `1e6d7e2d` | 23020136 | 2026-03-07 | 26 | 27 | +1 |
| `4cd5b7f7` | 33660591 | 2026-03-09 | 26 | 27 | +1 |
| `5b6d0c3b` | 33660611 | 2026-03-10 | 26 | 27 | +1 |
| `6edfeada` | 23020136 | 2026-03-11 | 26 | 27 | +1 |
| `7cc7af88` | 33660615 | 2026-03-11 | 26 | 27 | +1 |
| `85fa8e34` | 33660689 | 2026-03-08 | 26 | 27 | +1 |

**Code-Level Root Cause:**

This is a direct consequence of **Root Cause 1**. Because RFIS starts window generation 10 minutes earlier than OS (first midpoint at 9:50 AM vs 10:00 AM), RFIS can fit **one additional window** at the beginning of the schedule day while both systems end at the same closing time.

With 20-minute intervals and a ~9-hour store operating window (e.g., 10:00 AM – 6:40 PM):
- OS: 26 windows starting at 10:00 AM midpoint
- RFIS: 27 windows starting at 9:50 AM midpoint (one extra at the start)

The comparison function `compareDeliveryAvailabilityResults` in `DeliveryOptionsUtil.kt` (line 508) is configured to **ignore exactly 1 day** with a different window count. However, when **multiple days** have ±1 differences, it correctly flags the mismatch.

### Root Cause 3: Today Sub-Second Truncation Offset

**Severity:** Very Low — cosmetic, typically < 1 minute offset
**Pattern:** On "today" entries, RFIS midpoints are clean minute-boundary times (e.g., `18:10:00Z`) while OS midpoints carry sub-second precision (e.g., `18:10:39.297Z`), producing offsets of −0.1 to −1.0 minutes.

**Evidence:**

| Trace ID (prefix) | Store | Today Midpoint (OS) | Today Midpoint (RFIS) | Offset (min) |
|---|---|---|---|---|
| `13785f51` | 23020128 | 18:10:39.297Z | 18:10:00Z | −0.65 |
| `1e6d7e2d` | 23020136 | 01:50:48.157Z | 01:50:00Z | −0.80 |
| `4cd5b7f7` | 33660591 | 02:30:38.014Z | 02:30:00Z | −0.63 |

**Code-Level Root Cause:**

**RFIS (`ScheduledDeliveryOption.kt`, `getRoundedUpTruncatedTime`):**
```kotlin
private fun getRoundedUpTruncatedTime(time: ZonedDateTime, roundUpMinutes: Int): ZonedDateTime {
    return if (time.minute.rem(roundUpMinutes) != 0) {
        time.plus(roundUpMinutes - time.minute.rem(roundUpMinutes).toLong(), MINUTES)
            .truncatedTo(MINUTES)  // Always truncates seconds/nanos
    } else {
        time.truncatedTo(MINUTES)  // Always truncates seconds/nanos
    }
}
```

RFIS **always** calls `.truncatedTo(ChronoUnit.MINUTES)`, zeroing out seconds and nanoseconds.

**OS (`DeliveryAvailabilityMapper.kt`, lines 3186–3188):**
```kotlin
val midpointTimestampRoundedUp = if (shouldTruncateWindowHours) {
    midpointTimestampRoundedUpPreTruncate.truncatedTo(ChronoUnit.MINUTES)
} else midpointTimestampRoundedUpPreTruncate
```

OS only truncates when the `shouldTruncateWindowHours` feature flag is **enabled**. When disabled, sub-second precision from the original `storeDayOperationalHour.start` timestamp is preserved.

**Impact:** This is a cosmetic difference. The sub-second offset does not affect the window start/end times shown to consumers, as display strings are formatted to minute granularity. The comparison function tolerates ±1 minute for asap ranges, but this offset still shows up in raw midpoint comparisons.

### Root Cause 4: Today ETA-Based First Window Offset

**Severity:** Moderate — shifts today's first window by +9 to +50 minutes
**Pattern:** On "today", RFIS's first scheduled window starts significantly later than OS's first window due to RFIS incorporating the ASAP ETA estimate.

**Evidence:**

| Trace ID (prefix) | Store | OS First Today Midpoint | RFIS First Today Midpoint | Offset (min) |
|---|---|---|---|---|
| `1880d74c` | 33660597 | ~18:00 | ~18:09 | +9 |
| `39a3e7f2` | 33660597 | ~18:00 | ~18:09 | +9 |
| `1e8fbdfa` | 23020122 | ~01:00 | ~01:49 | +49 |
| `3b1becc8` | 23020122 | ~01:00 | ~01:49 | +49 |
| `8e3e74d8` | 23020122 | ~23:50 | ~00:39 | +49 |
| `c0c93a2e` | 23020122 | ~23:50 | ~00:39 | +49 |

The +49 minute offset for store 23020122 is notably large, indicating a high ASAP ETA for that store.

**Code-Level Root Cause:**

**RFIS (`ScheduledDeliveryOption.kt`, `buildDeliveryWindows`):**
```kotlin
val asapStartTimeWithPadding = asapStartTime.plusMinutes(ASAP_START_TIME_PADDING_MINUTES)
val scheduleStartTimeWithPadding = scheduleStartTime.plusMinutes(SCHEDULE_START_TIME_PADDING_MINUTES)
val combinedStartTime = maxOf(asapStartTimeWithPadding, scheduleStartTimeWithPadding)
```

RFIS computes `combinedStartTime` as the **maximum** of:
1. `asapStartTime + ASAP_START_TIME_PADDING` — incorporates `expressTimeMaxRange` (ETA)
2. `scheduleStartTime + 5 minutes` — schedule start with padding

When the ASAP ETA is high (e.g., 40+ minutes from now), `asapStartTimeWithPadding` dominates, pushing the first scheduled window significantly later than the schedule start.

**OS (`DeliveryAvailabilityMapper.kt`, lines 3155–3160):**

For non-MIC BVIDs (like BVID 232), OS uses `storeDayOperationalHour.start` directly as the midpoint, **without incorporating ETA/ASAP time** for today's schedule calculation.

```kotlin
// Non-MIC path:
val midpointTimestamp = storeDayOperationalHour.start
```

OS only incorporates `standardWindowUpperBound` (ETA-derived) via the `shouldInitiateFromNextHour` function (line 5682), which rounds to the next full hour. This is a different mechanism from RFIS's `maxOf(asap, schedule)` approach.

**Impact:** For stores with high ASAP ETAs, RFIS may show significantly fewer "today" windows because the first window is pushed forward. Store 23020122's +49 minute offset means RFIS omits approximately 2–3 windows at the start of today's schedule.

### Root Cause 5: Day-Boundary Date Grouping Mismatch (Edge Case)

**Severity:** Low — rare edge case observed in 1 file
**Pattern:** Extreme midpoint offsets of **949.1 and 1430.0 minutes** on a single trace, caused by windows being grouped under different dates between OS and RFIS.

**Evidence:**

| Trace ID | Store | Day | OS Count | RFIS Count | Offsets (min) |
|---|---|---|---|---|---|
| `6edfeada` | 23020136 | 2026-03-10 | 2 | 0 | 949.1, 1430.0 |

**Context:** This request was made at **23:58 UTC** on March 9 (approximately 4:58 PM PDT). The extreme offsets (949 min ≈ 15.8 hours, 1430 min ≈ 23.8 hours) indicate that what OS grouped as "March 10" windows, RFIS either grouped under a different date or did not generate.

**Code-Level Root Cause:**

**OS (`DeliveryAvailabilityMapper.kt`, line ~3200):**
```kotlin
val midpointTimestampLocalized = midpointTimestampRoundedUpAfterSanitization
    .atZone(ZoneId.of(storeTimezoneId))
val dayDate = midpointTimestampLocalized.toProtoDate()
```

OS groups windows by the **localized midpoint date** (converting midpoint UTC → store timezone → date).

**RFIS (`DeliveryWindowBuilder.kt`):**
```kotlin
fun buildDate(zonedStartTime: ZonedDateTime): ProtoDate {
    return ProtoDate.newBuilder()
        .setYear(zonedStartTime.year)
        .setMonth(zonedStartTime.monthValue)
        .setDay(zonedStartTime.dayOfMonth)
        .build()
}
```

RFIS groups windows by the **start time** (not midpoint) in the store's local timezone. With a 20-minute interval and halfInterval of 10 minutes, `startTime = midpoint − 10 min`. Near day boundaries, `midpoint` might fall on day N while `startTime = midpoint − 10 min` falls on day N−1, causing the window to be bucketed under a different date.

At 23:58 UTC near a day boundary, the combination of different bucketing keys (midpoint vs start time) and potential timezone conversion differences can produce windows that appear on different dates in the two systems, resulting in the massive offset values observed.

**Impact:** This is an edge case affecting requests made very close to day boundaries (midnight UTC or local). It does not indicate incorrect window generation — just different date bucketing.

## What Needs To Be Done

### Priority 1: Resolve RC1 — Future-Day Midpoint Offset

This is the dominant mismatch affecting all future-day comparisons. To resolve:

1. **Investigate schedule hour inputs**: Verify whether `storeDayOperationalHour.start` (OS) and `scheduleStartTime` (RFIS) receive the same raw values from MDS. If they differ, align the data pipeline.

2. **Align padding logic**: Either:
   - **Option A**: Add `SCHEDULE_START_TIME_PADDING_MINUTES = 5` to OS before rounding, matching RFIS behavior.
   - **Option B**: Remove the 5-minute padding from RFIS to match OS behavior.
   - **Option C**: Accept the RFIS behavior as the intended target (since RFIS is the migration destination) and update the comparison tolerance.

3. **Align timezone rounding**: Choose whether rounding should use UTC (current OS) or store local timezone (current RFIS). The RFIS approach (local timezone) is arguably more correct for consumer-facing windows.

### Priority 2: Resolve RC4 — Today ETA-Based Offset

1. **Decision needed**: Should today's first scheduled window incorporate the ASAP ETA? RFIS does this (`maxOf(asap, schedule)`), OS does not for non-MIC BVIDs.
   - If RFIS's behavior is correct (first scheduled window should not overlap with ASAP delivery time), no RFIS change needed — update OS comparison tolerance or accept RFIS behavior post-migration.
   - If OS behavior is correct, remove the `maxOf` logic from RFIS for non-MIC verticals.

### Priority 3: Resolve RC3 — Truncation Difference

1. **Enable `shouldTruncateWindowHours`** in OS for BVID 232, or accept that RFIS's always-truncate behavior is the correct target post-migration.

### Priority 4: Resolve RC5 — Date Grouping Edge Case

1. **Align bucketing key**: Decide whether to group windows by midpoint date (OS) or start-time date (RFIS). This only matters for windows near day boundaries.
2. This is low priority as it affects < 1% of requests and only at specific times of day.

### Shadow Read Comparison Tolerance Updates

If the decision is to accept RFIS behavior as-is (since RFIS is the migration target):
- Update `compareDeliveryAvailabilityResults` in `DeliveryOptionsUtil.kt` to:
  - Tolerate ±1 window count on **all** days (not just 1 day)
  - Tolerate ±10 minute midpoint offsets on future days
  - Tolerate sub-second truncation differences
  - Tolerate ETA-based offset on today

## Conclusion

The shadow read comparison for **BVID 232** reveals **5 distinct root causes** for mismatches between OS and RFIS, all stemming from architectural differences in window computation logic rather than data errors or bugs.

The **dominant mismatch** is a consistent **−10 minute midpoint offset on future days** (RC1), caused by RFIS's 5-minute schedule start padding combined with UTC vs local timezone rounding divergence. This single root cause cascades into the **±1 window count differences** (RC2) observed on 8 files.

For **today's windows**, two additional root causes appear: **sub-second truncation** (RC3, cosmetic) and **ETA-based first window offset** (RC4, moderate impact for high-ETA stores). A single **day-boundary edge case** (RC5) accounts for the remaining anomaly.

**94.8% of all requests (293/309)** produced zero scheduled windows in both systems and showed no mismatch. The mismatches are limited to the **5.2% of requests** where scheduled delivery was available.

**Key decision for migration**: Since RFIS is the migration target, the team should evaluate whether RFIS's window generation behavior (with padding, local timezone rounding, and ETA incorporation) represents the **desired** consumer experience post-migration. If so, the primary action is to update comparison tolerances to reduce false-positive mismatch alerts during the transition period, and then proceed with cutover.
